## 1. Bug 修复

### 1.1 训练输出重复（每行显示两次）

**文件**: `gui/workers.py`

**问题**: 控制台每行输出都出现两次。

**根因**: `TrainWorker.run()` 中做了双重捕获：
- 替换 `sys.stdout` 为 `EmitCapturer`（LOGGER 的 StreamHandler 已通过 stdout 流向 capturer）
- 又向 LOGGER 添加了额外的 `LogHandler`（也写入同一个 capturer）

**修复**: 删除多余的 `LogHandler`，只依赖 `EmitCapturer` 捕获 stdout/stderr。

### 1.2 第二次训练 mAP/precision 图表空白

**文件**: `gui/workers.py`

**问题**: 第一次训练图表正常，从第二次训练开始 mAP 和 precision 图表空白。

**根因**: ultralytics LOGGER 的 `StreamHandler` 在模块加载时创建，存储了当时 `sys.stdout` 的对象引用。第一次训练时模块导入，StreamHandler 指向第一个 capturer。第二次训练创建新 capturer，但 LOGGER 的 handler 仍指向旧 capturer → 旧 worker 的 `_cur_epoch=10` → 去重跳过 → 验证数据不发射。

**修复**: 每次训练开始时，更新 LOGGER 的 StreamHandler 指向新 capturer：
```python
ultralytics_logger = logging.getLogger("ultralytics")
for h in ultralytics_logger.handlers:
    if isinstance(h, logging.StreamHandler) and hasattr(h, 'stream'):
        h.stream = capturer
```

### 1.3 训练输出信号洪水导致栈溢出

**文件**: `gui/workers.py`

**问题**: 训练过程中进程崩溃，退出码 `0xC00000FD`（Windows 栈溢出）。

**根因**: TQDM 进度条每秒用 `\r` 更新几十次，每次触发 `_parse_metrics()` 匹配训练行后发射 `epoch_done` 信号，信号洪水导致 Qt 事件队列溢出。同时在 `_parse_metrics` 中使用 `print()` 调试导致无限递归（stdout 被捕获 → 触发解析 → 再次 print → 递归）。

**修复**:
- 训练 loss 只在值变化时发射（去重）
- 验证指标用 `_last_val_epoch` 追踪，每 epoch 只发射一次
- 禁止在 `_parse_metrics` 中使用 `print()`

### 1.4 Force Stop 按钮无法及时终止训练

**文件**: `gui/workers.py` + `gui/main_window.py`

**问题**: 点击 Stop 后训练继续运行，要等到当前 epoch 结束才生效。

**根因**: 原来的 `on_train_epoch_start` 回调只在每个 epoch 开始时检查 stop 标志。

**修复**:
- 删除 Pause 按钮，Stop 改为 Force Stop
- 添加 `on_train_batch_end` 回调，在每个 batch 结束后检查 stop 标志
- `force_stop()` 方法替代 `pause()` / `resume()` / `stop()`

### 1.5 训练完成后 GPU 内存未释放

**文件**: `gui/workers.py` + `gui/main_window.py`

**问题**: 训练完成后 `nvidia-smi` 显示显存占用与训练时相同。

**根因**: `model`（YOLO 对象）和 `result`（训练结果）在线程局部变量中被引用。YOLO 内部存在循环引用（model ↔ trainer），普通引用计数无法回收。

**修复**: 训练结束后显式清理：
```python
del model, result
import gc
gc.collect()                    # 强制回收循环引用
if torch.cuda.is_available():
    torch.cuda.empty_cache()    # 让 CUDA 释放未使用的显存块
```

同时在主线程的 `_on_train_done()` / `_on_train_error()` 中也添加清理作为双重保障。

### 1.6 第二次训练状态未重置

**文件**: `gui/workers.py`

**问题**: 第二次训练时 `_cur_epoch` 保持上一次训练的最终值。

**根因**: `run()` 结束时 `_cur_epoch` 未重置。

**修复**: 在训练完成后（`model.train()` 返回后）重置：
```python
self._cur_epoch = None
self._last_val_epoch = -1
```

## 3. 技术实现

### 3.1 训练输出捕获

```
ultralytics 训练输出
    ↓
sys.stdout (EmitCapturer)
    ↓ _parse_metrics()
训练 loss: 值变化时即时 emit（TQDM \r 去重）
验证 mAP:  每 epoch 去重 emit（_last_val_epoch）
学习率:    on_train_epoch_start 回调
    ↓
epoch_done 信号 → GUI 图表更新
```

- LOGGER 的 StreamHandler 在每次训练开始时更新指向新的 capturer
- `_parse_metrics` 中禁止 `print()`（stdout 被捕获会无限递归）

### 3.2 图表更新机制

```
_train_pending_metrics (缓冲队列)
        ↓ QTimer 500ms
_flush_charts() (主线程)
        ↓
_apply_metrics() → 更新 6 个 matplotlib 图表
_update_time_display() → 更新时间显示
```

### 3.3 Force Stop 流程

```
Force Stop 按钮
    → self.stop_flag = True
    → on_train_batch_end 回调检查
    → trainer.stop = True
    → 当前 batch 完成后 break
    → validate() + save_model()
    → final_eval() + strip_optimizer()
    → del model + gc.collect() + cuda.empty_cache()
    → 正常退出
```

不是杀进程，而是设置标志让训练循环自己走完清理流程。模型文件完整且已压缩（optimizer state 被移除）。

### 3.4 GPU 内存管理

训练结束后在 worker 线程中：
```python
del model, result          # 删除 Python 引用
gc.collect()               # 强制垃圾回收，释放循环引用
torch.cuda.empty_cache()   # 让 CUDA 释放缓存的显存块
```

主线程 `_on_train_done()` / `_on_train_error()` 中也执行清理作为双重保障。

### 3.5 close_mosaic

`close_mosaic`（默认 10）控制最后 N 个 epoch 关闭 Mosaic 数据增强。关闭时会重建 dataloader，暂停几秒属正常现象。

| 场景 | close_mosaic | 说明 |
|------|-------------|------|
| 默认 | 10 | 推荐，最后 10 epoch 关 Mosaic |
| 小数据集 | 5 | 早点关让模型收敛 |
| 大数据集 | 15~20 | 多练复杂场景 |
| 关闭 | 0 | 全程 Mosaic，一般不建议 |

### 3.6 模型缩放选择器

Train 页面提供 `n/s/m/l/x` 缩放选择器，仅对 `.yaml` 架构文件生效：

```python
# 用户选择 yolo26.yaml + scale=n
# → 自动拼接为 yolo26n.yaml
# → ultralytics 内部加载 yolo26.yaml 并应用 n 缩放系数
```

- `.pt` 文件：缩放选择器不生效，直接加载预训练权重
- `.yaml` 文件：自动拼接缩放后缀（如 `yolo26.yaml` + `n` → `yolo26n.yaml`）
- 已有后缀的文件不会重复添加（如 `yolo26n.yaml` + `n` → 不变）

### 3.7 配置文件

所有可调参数集中在 `gui/settings.py`，`main_window.py` 通过 `from . import settings` 导入使用。
